1-2 — Semantic HTML & Accessibility

~60 minutes · Week 1, Day 1 · Editor and browser only — no terminal, no git yet

Why this matters: the profile page you start here grows into your portfolio project at the end of this week. Semantic HTML is also a favorite interview topic.

Idea 1 — Tags that say what they mean

freeCodeCamp taught you the tags; the professional habit is choosing them for meaning. <div class="header"> and <header> look identical on screen — but only the second one is announced as a landmark by a screen reader, understood by search engines, and instantly legible to the next developer.

The skeleton to internalize — six tags plus one rule:

<header>   who/what this page is; usually contains <nav>
<nav>      the page's main navigation links
<main>     the unique content — exactly ONE per page
<section>  a thematic grouping inside main
<article>  self-contained content that would make sense alone
<footer>   closing info: copyright, contact, small links

<aside> (tangential content, e.g. a sidebar) exists too — it's on the cheat sheet; look it up when you need it, don't memorize it now.

Idea 2 — Accessibility is checkable, not vibes

Semantic structure is the first half of accessibility. The second half is verifying it: browsers ship an auditor called Lighthouse that scores your page and lists concrete fixes. You'll run it on your own page in a few minutes.

Do it for real (~35 min)

All in your editor and browser. (If you don't have an editor yet, install VS Code first — ask your teacher for help.)

  1. In your file manager or editor, create the folder ~/code/week-1 (a code folder in your home folder, with week-1 inside). You'll reuse it all week. Success check: in VS Code, File → Open Folder shows week-1 empty and open.
  2. Create a new file in it named index.html. You should see the file appear in the editor's sidebar.
  3. Type (don't paste) the base document: 
    <!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>My profile</title>
</head>
<body>

</body>
</html>
    Success check: open the file in your browser (double-click it, or drag it onto a browser window). You see an empty page whose tab reads "My profile".
  4. Inside <body>, add a <header> containing an <h1> with your name. Save, refresh the browser. You should see your name.
  5. Inside the header, below the <h1>, add a <nav> with three links: <a href="#about">About</a>, likewise #projects and #contact. Refresh. You should see three links under your name.
  6. After the header, add <main> containing one <section id="about"> with an <h2>About</h2> and a paragraph about you. Refresh. You should see the About block under your nav.
  7. Still inside <main>, add a <section id="projects"> with an <h2>Projects</h2> and two <article>s, each holding an <h3> project name and a one-line description (freeCodeCamp projects count). Refresh. You should see both project blocks below About.
  8. Close the page with a <footer id="contact"> containing your email or GitHub link. Refresh. Success check: clicking each nav link scrolls/jumps to its section.
  9. Audit it — in Chrome or Chromium (Lighthouse ships in Chromium-based browsers; Firefox doesn't have it — if Firefox is your daily browser, open the page in Chromium just for audits): press F12 → find the Lighthouse tab (it may hide behind ») → check only Accessibility → Analyze. You should see a score. Read the top suggestion — fix it if it's one you understand, ask your teacher if not.

Check yourself — fix the snippet

This page fragment renders fine but fails semantics review. Find all three problems before revealing:

<div class="header"><h1>Salma's Recipes</h1></div>
<main>
  <section><h2>Latest</h2>...</section>
</main>
<main>
  <section><h2>Popular</h2>...</section>
</main>
<div class="footer">© 2026</div>
Reveal the three problems
  1. <div class="header"> should be <header> — a div announces nothing to assistive tech.
  2. Two <main> elements — a page gets exactly one; merge the sections into it.
  3. <div class="footer"> should be <footer>.
