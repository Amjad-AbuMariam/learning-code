Lesson 1 — The Branch & PR Workflow

~40 minutes · Curriculum: Week 1 (Fundamentals & Toolkit)

Why this, first: freeCodeCamp taught you to code; it didn't teach you to work like a developer. In any job, code reaches production through branches and pull requests. Every later week of the GSG curriculum assumes this loop is muscle memory. Today you make it real by doing it once, end to end, on a repo you create.

The one idea

main is sacred: it always works, and you never commit to it directly. All work happens on a short-lived branch, which reaches main only through a pull request — a reviewable diff. That's the whole professional workflow; everything else is detail. (Full command list: cheat sheet.)

A branch in git is just a 41-byte file containing a commit hash — creating one is instant and free. That's why pros branch for everything, even a one-line fix.

Do it for real (~25 min)

Work through these in your terminal. Check each box as you go.

  1. Create a repo on GitHub called workflow-practice (add a README so it isn't empty), then git clone it and cd in.
  2. Run git switch -c feat/add-bio. Run git status — notice it names your new branch.
  3. Edit README.md: add two lines about yourself.
  4. Stage deliberately: git add -p, answer y to the hunk. Then git diff --staged to review exactly what will be committed.
  5. Commit: git commit -m "Add bio so visitors know whose repo this is" — imperative mood, says why.
  6. Publish: git push -u origin feat/add-bio.
  7. On GitHub, open a pull request from your branch into main. Read the diff view — this is what a reviewer sees.
  8. Merge the PR on GitHub, then locally: git switch main && git pull && git branch -d feat/add-bio.
  9. Run git log --oneline --graph and find your merge in the history.
