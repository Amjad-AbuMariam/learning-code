1-10 — Week Project: Portfolio

Days 4–5 (this is where most of the week's hours go) · Source: curriculum portfolio brief + accessibility & responsiveness assignment

Why this matters: this is the first portfolio piece of the mission. Everything the week taught gets used together, under the professional workflow, reviewed like real code.

The brief

Build your ePortfolio as a multi-section site. From the curriculum's brief, it needs these five parts (pages or sections — your call as designer):

  1. About me — who you are, current status, aspirations. Your week-1 profile page is the seed; grow it, don't start over.
  2. Skills & projects — cards linking your work (freeCodeCamp projects count today; this course will replace them fast).
  3. CV — a link or embedded summary.
  4. Reflections — what you've learned; this week is your first entry.
  5. Inspiration — people, sites, quotes that pull you forward.

Requirements (what "done" means)

  • Semantic skeleton throughout — exactly one <main> per page, landmarks for header/nav/footer (lesson 1-2)
  • Layout with flexbox (1-3), responsive mobile-first with at least one media query (1-6), viewport tag present
  • Lighthouse accessibility score ≥ 90, and you understand every remaining warning (1-2)
  • All work through the git loop (1-5): a GitHub repo named portfolio, every feature on a branch, merged by PR — push repeatedly and frequently, as the assignment says; no day-5 mega-commit
  • At least one <script> using this week's JS: an array of projects rendered with map/join, or the skill counter from 1-9 grown up

How to work (~2 days)

  1. Design first (Day 3's UX slot): sketch the layout before coding — Figma if you want the industry tool (curriculum's starter playlist), paper if you don't. Alongside it, write three one-line user stories for your visitors — they decide your nav order. Success check: you can point at where each of the five parts lives and which story it serves, before writing a line.
  2. Create the portfolio repo on GitHub with a README. In a terminal: cd ~/code, git clone + the repo's URL (1-5 exercise 3 form), cd portfolio. Copy your page in as the starting point: cp ~/code/week-1/index.html ~/code/portfolio/. Check: ls shows index.html and README.md. Commit and push the baseline on main — the one exception to the branch rule: git add index.html, git commit -m "Import week-1 profile page as baseline", git push. Check: the file is visible on GitHub.
  3. Work in vertical slices, one branch + PR each: nav-and-layout, about-section, projects-section, … Merge each before starting the next. Check after each merge: in the terminal, from ~/code/portfolio: git switch main, git pull, then open index.html in the browser — the merged state always works.
  4. Run Lighthouse (F12 → Lighthouse tab in Chromium, exactly as in lesson 1-2) on your page after every merged slice, not at the end. Fixing accessibility incrementally is minutes; retrofitting it is days.
  5. Get reviewed. The curriculum's code review guide: reviewers read your repo, raise issues, phrase suggestions as questions. Ask your teacher (the agent) to review each PR before you merge it — and answer its questions in the PR conversation, don't just fix silently. Etiquette works both ways.

Check yourself

Before calling the week done, close the loop from memory — no cheat sheet: branch → edit → stage → commit → push → PR → merge → sync → delete branch. If any step needed a lookup, tell your teacher; that's the retrieval signal it watches for.
