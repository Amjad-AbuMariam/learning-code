1-6 — Responsive Design

~45 minutes · Week 1, Day 2 · Source: curriculum Day 2 Responsive Design workshop

Why this matters: the Week 1 project is judged on desktop and mobile views. Most of your visitors — and every interviewer who opens your portfolio on a phone — see the mobile layout first.

Idea 1 — One page, many screens

A responsive page adapts its layout to the screen instead of assuming a desktop. Two ingredients you already half-have:

  1. The viewport meta tag — without it, phones render a zoomed-out desktop page: 
    <meta name="viewport" content="width=device-width, initial-scale=1">
  2. Layouts that flex — your flex-wrap from lesson 1-3 is already responsive behavior.

Idea 2 — Media queries, mobile-first

A media query applies CSS only when a condition holds:

/* base rules: written for small screens first */
nav { flex-direction: column; }

/* from 40rem wide and up, override */
@media (min-width: 40rem) {
  nav { flex-direction: row; }
}

Mobile-first means the plain rules serve the smallest screen and min-width queries add complexity as space grows — simpler than desktop-first with max-width overrides, because the base case is the simple layout. (BEM naming and deeper responsive patterns are this week's research-afternoon topics.)

Do it for real (~25 min)

Two threads from here on: the CSS work, and putting ~/code/week-1 under git — like everything you'll ever build from now on.

  1. Open a terminal: cd ~/code/week-1 (confirm with pwd), then git init. You should see "Initialized empty Git repository…".
  2. Snapshot the page as it stands: git add index.html, then git commit -m "Profile page from lessons 1-2 and 1-3". Check: git status now says "nothing to commit, working tree clean".
  3. Add the viewport meta tag inside <head>, next to <meta charset>. Success check: press F12, click the phone/tablet icon (device toolbar), pick a phone — text is readable, not a zoomed-out miniature.
  4. Make the nav mobile-first: change your nav rule to flex-direction: column; (keep display: flex; gap: 1rem;). Refresh in phone view. You should see the links stacked vertically.
  5. Add the desktop override at the end of your <style> block: 
    @media (min-width: 40rem) {
  nav { flex-direction: row; justify-content: flex-end; }
}
    Success check: drag the device-toolbar width (or toggle it off): links stack when narrow, become a right-aligned row when wide.
  6. Give the project cards a minimum dignity: add min-width: 15rem; to .cards article. You should see (in phone view) the cards refuse to squeeze thinner and wrap into a column instead — flex-wrap from 1-3 doing the work.
  7. Commit: git add index.html, git commit -m "Make profile page responsive". Check: git log --oneline shows both commits.
  8. Publish the repo — from memory of exercise 2 in lesson 1-5: create an empty week-1 repository on GitHub (no checkboxes), then, in the same terminal (confirm pwd still ends in /code/week-1), run the three "push an existing repository" lines GitHub shows. Success check: your page's code is visible on GitHub. Lesson 1-9 opens a pull request against this very repo.

Check yourself — predict the output

Screen is 30rem wide. Base rule says nav { flex-direction: column; }; a @media (min-width: 40rem) block says row. Which applies, and what if the screen grows to 45rem?

Reveal

At 30rem the query condition is false, so the base column applies. At 45rem the query matches and its row rule wins because it comes later in the stylesheet. Mobile-first in one sentence: base = small screen, queries = progressively wider.
