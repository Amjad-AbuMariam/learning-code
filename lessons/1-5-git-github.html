1-5 — Git & GitHub

~90 minutes · Week 1, Day 2 · Source: curriculum git & GitHub workshop (exercises 1–3)

Why this matters: in any developer job, code reaches production through commits, branches, and pull requests. This lesson makes that loop real, three small exercises at a time — and from today, everything you build in this course is committed.

Idea 1 — Git takes snapshots; GitHub hosts them

Git is a version control system that runs on your machine: it records commits — snapshots of your project with a message. GitHub is a website that hosts a copy of your git history (a repository) so it's backed up, shareable, and reviewable. Git = local; GitHub = remote.

Idea 2 — main is sacred

The main branch always works; you never commit to it directly. Work happens on a branch, which reaches main only through a pull request. Everything else in the workflow is detail around that rule.

Exercise 0 — Introduce yourself to git (once ever, ~5 min)

  1. In a terminal — any directory works, this setting is global — tell git who you are (it signs your commits): 
    git config --global user.name "Your Name"
git config --global user.email "you@example.com"
    Success check: git config --list prints both values (press q if the list opens in a scroller).

Exercise 1 — Local only: the snapshot cycle (~20 min)

From the workshop's exercise 1, in your terminal:

  1. Create and enter the repository: 
    cd ~/code
mkdir my-first-git-repo
cd my-first-git-repo
pwd                  # ends in my-first-git-repo
    Check: pwd ends in my-first-git-repo.
  2. Initialize it and inspect hidden files: 
    git init              # Initialized empty Git repository…
ls -a                 # shows the .git folder
    You should see "Initialized empty Git repository…". .git is the entire history database.
  3. Create a file and inspect its status: 
    touch index.html
git status             # index.html is red under Untracked files
    You should see index.html in red under "Untracked files" — git sees it but isn't recording it yet.
  4. Stage the file and inspect status again: 
    git add index.html
git status             # index.html is green under Changes to be committed
    You should see it turn green under "Changes to be committed" — the staging area.
  5. Commit the staged file: 
    git commit -m "Add index.html"
    You should see a summary line with the message and "1 file changed".
  6. Inspect the history: 
    git log
    You should see your commit with author, date, message. If the view takes the whole screen, it's a scroller: ↑/↓ to move, q to quit.
  7. Repeat the cycle twice on your own, exactly as the workshop asks: (a) create any second file, stage, commit; (b) delete index.html — run ls first to confirm what you're removing (the 1-4 ritual; safe here anyway, a committed file can always be restored by git), then run: 
    rm index.html
git add index.html
git commit -m "Delete index.html"
    (Yes — you stage a deletion too.) Success check: git log shows three commits, newest first.

Exercise 2 — Connect to GitHub (~20 min)

  1. On github.com (create an account first if needed — ask your teacher if login/authentication gives you trouble; that's normal the first time), click + → New repository, name it my-first-git-repo, and leave every checkbox unchecked (no README), exactly as the workshop says. You should see a "…or push an existing repository from the command line" instructions page.
  2. Copy the three lines GitHub shows under "push an existing repository" and run them in your repo folder: 
    git remote add origin …   # name the remote
git branch -M main        # rename the branch
git push -u origin main   # upload
    The push asks you to authenticate — what you'll see depends on your setup: either a browser window opens asking you to authorize, or the terminal prompts for your GitHub username and a token. If the terminal asks for a "password", stop there: GitHub no longer accepts account passwords on the command line — ask your teacher to set up SSH keys or a token with you (five minutes, once ever).
  3. Refresh the repository page on GitHub. Success check: your files and all three commit messages are visible online.

Exercise 3 — Branch, pull request, merge (~30 min)

  1. On GitHub: create a new repository called workflow-practice, this time with a README checked.
  2. On the repo page, click the green Code button and copy the URL. Then locally run (the clone line ends up like git clone https://github.com/yourusername/workflow-practice.git): 
    cd ~/code
git clone URL                 # replace URL with what you copied
cd workflow-practice
ls                            # shows README.md
    You should see a short clone progress report, then README.md in the listing.
  3. Create and switch to a branch: 
    git switch -c add-hello-page
    You should see "Switched to a new branch". Coursebook note: the workshop writes git checkout -b; same effect, legacy spelling.
  4. Create hello.html containing <h1>Hello world</h1> (editor or echo — your choice now). Stage and commit it: 
    git add hello.html
git commit -m "Add hello page"
  5. Publish the branch: 
    git push -u origin add-hello-page
    You should see the push output end with a GitHub URL for creating a pull request.
  6. Open that URL (or the yellow "Compare & pull request" banner on the repo page). Read the Files changed tab — this diff is exactly what a reviewer sees. Click Create pull request.
  7. Click Merge pull request → Confirm, then Delete branch (the button GitHub offers right after merging — remote branches are deleted from here). You should see the PR marked purple "Merged".
  8. Sync your machine and delete the local branch: 
    git switch main
git pull
git branch -d add-hello-page
    You should see hello.html arrive. Success check: git log --oneline shows the merge on main, and git branch lists only main.
  9. The workshop says: redo the loop once more, styling the page — branch, edit, commit, push, PR, merge, sync, delete both branches. Do it from memory; peek at the cheat sheet only when stuck. Success check: a second merged PR on GitHub, and git branch again lists only main.
