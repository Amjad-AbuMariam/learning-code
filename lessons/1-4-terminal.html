1-4 — Terminal

~60 minutes · Week 1, Day 2 · Source: curriculum terminal workshop · No prior terminal experience assumed

Why this matters: the next lesson (git) and everything from Node onward lives in the terminal. One hour here removes friction from the entire course.

Idea 1 — Terminal vs shell, and the shape of a command

The terminal is the window; the shell (yours is bash) is the program inside it that reads a command, runs it, and prints the result. Every command has the same shape:

ls        -l        code
│         │         │
command   flag      argument
(what)    (how)     (on what)
Two habits that make beginners look experienced: press Tab to autocomplete paths (if nothing completes, your path is wrong before you run it), and press ↑ to recall previous commands.

Idea 2 — You are always somewhere

All files live in one tree that starts at the root. Your shell always sits at one spot in that tree — the working directory. Three shorthands do most of the work: ~ is your home folder, . is here, .. is one level up. A path starting with / is absolute (works from anywhere); anything else is relative to where you are.

Do it for real (~40 min)

Type every command — no copy-paste; typing builds the recall. Predict each output before Enter.

Part A — Orient (pwd, ls, cd)

  1. Open a terminal. Type pwd and press Enter. You should see your home folder's absolute path — something like /home/yourname. (Check with whoami if unsure of your username.)
  2. Type ls. You should see your home folders, including the code folder from lesson 1-2.
  3. Type ls -a. You should see extra entries starting with a dot — hidden files. (You'll meet .git this way tomorrow.)
  4. Type cd code/week-1 then pwd. You should see the path ending in /code/week-1, and ls shows index.html — the file you built in lessons 1-2/1-3, reached from the terminal side.
  5. Go home three ways, running pwd after each way. Way 1: cd .. then cd .. again (relative, one level at a time — you land home). Way 2: cd ~/code/week-1 to jump back down, then plain cd (no argument always means home). Way 3: cd ~/code/week-1 again, then cd ~. Success check: you can predict each pwd output before pressing Enter.

Part B — Create and inspect (mkdir, touch, echo, cat)

  1. From home: mkdir -p code/terminal-practice, then cd code/terminal-practice. You should see pwd ending in terminal-practice. (-p creates parents as needed and doesn't complain if they exist.)
  2. touch notes.md then ls. You should see notes.md.
  3. A caution before the next command: > replaces a file's entire content, so know what's in a file before redirecting into it — here notes.md is the empty file you just created, so nothing can be lost. Now: echo "terminal day one" > notes.md then cat notes.md. You should see the line printed back. >> appends instead: echo "second line" >> notes.md, cat notes.md — two lines now.

Part C — Move, copy, delete (mv, cp, rm)

  1. cp notes.md backup.md then ls. You should see both files.
  2. mv backup.md old-notes.md then ls. You should see the file renamed — mv to a new name is how you rename.
  3. mkdir archive, mv old-notes.md archive, ls archive. You should see old-notes.md inside — same command, moving instead of renaming. No success message is normal: in the shell, no news is good news.
  4. Deletion, carefully. rm has no trash can and no undo. The safe ritual: ls first, read what you're about to remove, then delete. Run ls archive, then rm archive/old-notes.md, then ls archive — empty now. The curriculum's safer alternative when unsure: gio trash FILE sends it to the desktop trash instead.
  5. Clean up the whole practice folder: cd ~, ls code (confirm terminal-practice is what you think it is), then rm -r code/terminal-practice (-r = recursive, needed for folders). Success check: ls code no longer lists it, and week-1 is untouched.
You'll eventually meet rm -rf in the wild. The curriculum's own warning applies: it deletes without asking and without mercy — never run it on a path you haven't just ls'd, and never anywhere near /. When in doubt, gio trash.

More commands (man, xdg-open, cp -r, quoting names with spaces) live on the terminal cheat sheet — look up, don't memorize.

Check yourself — predict the output

You are in your home folder (where step 13 left you — confirm with pwd). What does cd code/terminal-practice try to do, and what happens right now?

Reveal

It's a relative path: from home, into code, then into terminal-practice — which you deleted in step 13, so bash prints No such file or directory. Errors like this are navigation feedback, not failures.
