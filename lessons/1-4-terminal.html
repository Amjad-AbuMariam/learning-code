1-4 — Terminal

~60 minutes · Week 1, Day 2 · Source: curriculum terminal workshop · No prior terminal experience assumed

Why this matters: the next lesson (git) and everything from Node onward lives in the terminal. One hour here removes friction from the entire course.

Idea 1 — Terminal vs shell, and the shape of a command

The terminal is the window; the shell (yours is bash) is the program inside it that reads a command, runs it, and prints the result. Every command has the same shape:

ls        -l        code
│         │         │
command   flag      argument
(what)    (how)     (on what)
Two habits that make beginners look experienced: press Tab to autocomplete paths (if nothing completes, your path is wrong before you run it), and press ↑ to recall previous commands.

Idea 2 — You are always somewhere

All files live in one tree that starts at the root. Your shell always sits at one spot in that tree — the working directory. Three shorthands do most of the work: ~ is your home folder, . is here, .. is one level up. A path starting with / is absolute (works from anywhere); anything else is relative to where you are.

Do it for real (~40 min)

Type every command — no copy-paste; typing builds the recall. Predict each output before Enter.

Part A — Orient (pwd, ls, cd)

  1. Open a terminal. Type these commands: 
    pwd          # your home folder's absolute path
whoami       # your username, if unsure
    You should see your home folder's absolute path — something like /home/yourname.
  2. Type this command: 
    ls           # list your home folders
    You should see your home folders, including the code folder from lesson 1-2.
  3. Type this command: 
    ls -a         # include hidden files
    You should see extra entries starting with a dot — hidden files. (You'll meet .git this way tomorrow.)
  4. Type these commands: 
    cd code/week-1   # move into the week-1 folder
pwd              # show the current path
ls               # list the profile page
    You should see the path ending in /code/week-1, and ls shows index.html — the file you built in lessons 1-2/1-3, reached from the terminal side.
  5. Go home three ways, running pwd after each way: 
    cd ..
cd ..
pwd              # Way 1: relative, one level at a time
cd ~/code/week-1
cd
pwd              # Way 2: plain cd means home
cd ~/code/week-1
cd ~
pwd              # Way 3: ~ means home
    Success check: you can predict each pwd output before pressing Enter.

Part B — Create and inspect (mkdir, touch, echo, cat)

  1. From home, type: 
    mkdir -p code/terminal-practice   # create the practice folder and parents
cd code/terminal-practice          # enter it
pwd                                # show the current path
    You should see pwd ending in terminal-practice. (-p creates parents as needed and doesn't complain if they exist.)
  2. Type: 
    touch notes.md   # create an empty file
ls               # list the file
    You should see notes.md.
  3. A caution before the next command: > replaces a file's entire content, so know what's in a file before redirecting into it — here notes.md is the empty file you just created, so nothing can be lost. Type: 
    echo "terminal day one" > notes.md   # replace the empty file
cat notes.md                          # print its contents
echo "second line" >> notes.md       # append instead
cat notes.md                          # print both lines
    You should see the line printed back, then two lines now.

Part C — Move, copy, delete (mv, cp, rm)

  1. Type: 
    cp notes.md backup.md   # copy the file
ls                      # list both files
    You should see both files.
  2. Type: 
    mv backup.md old-notes.md   # rename the file
ls                           # list the renamed file
    You should see the file renamed — mv to a new name is how you rename.
  3. Type: 
    mkdir archive              # create a destination folder
mv old-notes.md archive     # move the file into it
ls archive                  # list the destination
    You should see old-notes.md inside — same command, moving instead of renaming. No success message is normal: in the shell, no news is good news.
  4. Deletion, carefully. rm has no trash can and no undo. The safe ritual: ls first, read what you're about to remove, then delete. Run: 
    ls archive                    # confirm the file to remove
rm archive/old-notes.md       # delete it permanently
ls archive                    # verify it is empty
    Empty now. The curriculum's safer alternative when unsure: gio trash FILE sends it to the desktop trash instead.
  5. Clean up the whole practice folder. Confirm terminal-practice is what you think it is before the destructive command: 
    cd ~
ls code                         # verify the folder before deleting
rm -r code/terminal-practice    # -r = recursive, needed for folders
ls code                         # verify cleanup; week-1 is untouched
    Success check: ls code no longer lists it, and week-1 is untouched.
You'll eventually meet rm -rf in the wild. The curriculum's own warning applies: it deletes without asking and without mercy — never run it on a path you haven't just ls'd, and never anywhere near /. When in doubt, gio trash.

More commands (man, xdg-open, cp -r, quoting names with spaces) live on the terminal cheat sheet — look up, don't memorize.

Check yourself — predict the output

You are in your home folder (where step 13 left you — confirm with pwd). What does cd code/terminal-practice try to do, and what happens right now?

Reveal

It's a relative path: from home, into code, then into terminal-practice — which you deleted in step 13, so bash prints No such file or directory. Errors like this are navigation feedback, not failures.
