Git Branch & PR Workflow

The loop you'll run on every feature, forever.

The cycle

git switch main
git pull                      # start from the latest main
git switch -c feat/short-name # new branch per unit of work
# ...edit files...
git add -p                    # stage hunk by hunk, review as you go
git commit -m "Add X so that Y"
git push -u origin feat/short-name
# open PR on GitHub → review → merge
git switch main && git pull   # sync, then delete the branch
git branch -d feat/short-name

Imperative mood, says why not just what: Add validation so empty forms can't submit — not fixed stuff.

Glossary

TermMeaning

mainThe shared, always-working branch; never commit to it directly
remote / originThe copy of the repo on GitHub
pull request (PR)A request to merge your branch, with a diff others can review
mergeCombine one branch's commits into another
staging areaThe set of changes queued for the next commit
